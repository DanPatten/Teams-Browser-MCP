Specification: Teams Browser MCP
1. Overview

A background Windows application residing in the system tray, responsible for securely handling user authentication for the headless Microsoft Teams MCP server. Because enterprise Single Sign-On (SSO) and Multi-Factor Authentication (MFA) cannot be easily automated, this utility provides a bridge by capturing a live, user-authenticated session and saving it for the headless MCP server to use.
2. Technology Stack

    Runtime: .NET 10

    Platform: Windows Desktop

    UI Framework: Windows Forms (WinForms) utilizing a custom ApplicationContext to run purely as a background system tray process without a persistent main window.

    Browser Engine: Microsoft Playwright (configured to leverage the local Chrome channel to mirror standard user behavior).

3. Application Flow & Lifecycle
Startup & Verification

    Silent Launch: The application starts in the background and places an icon in the Windows taskbar notification area.

    State Check: Upon launch, the app verifies the existence of a local authentication state file (e.g., .auth/state.json).

    First Run: If no state file exists, the application immediately triggers the Headed Authentication Sequence.

The Headed Authentication Sequence

    Launch: The application spawns a fully visible (headed) Chrome browser window and navigates to the Teams web client.

    User Interaction: The user manually completes their corporate login, including any required MFA challenges.

    DOM Monitoring: The background process monitors the browser's Document Object Model (DOM). It waits for a specific element that only renders after a successful login (such as the main chat layout container) with a generous timeout to allow for slow MFA approvals.

    Capture & Save: Once the successful login indicator is detected, the application extracts the browser context's storage state (containing all necessary session cookies and local storage tokens) and writes it to the local state file.

    Cleanup: The visible browser window automatically closes, and the application returns to a dormant monitoring state.

4. User Interface (System Tray)

The application interaction is limited entirely to the system tray to remain unobtrusive.

    Icon States: The tray icon will use visual indicators or tooltip text to communicate its current status (e.g., "Authenticating..." vs. "Ready").

    Context Menu: Right-clicking the tray icon provides a simple menu:

        Login/Logout - depending on state, if auth is expired then Re-Authenticate.

        Exit: Closes the background process.

5. Integration with the MCP Layer

    Session Sharing: The primary headless MCP server (which handles the actual reading and writing of messages) will be configured to boot its Playwright instance using the session state file generated by this tray application.

    Expiration Handling: Enterprise session tokens periodically expire. If the active MCP server detects a redirection to a login portal during a headless operation, it will return an authentication error to the LLM client. The user can then right-click the system tray icon and select "Re-Authenticate" to refresh the token.
